# Академия риэлторов: execution-plan

## 1. Цель

Собрать продаваемый и масштабируемый edtech-продукт, в котором:

- есть сильный публичный контур: лендинг, каталог, вход, checkout;
- есть рабочий внутренний контур: админ, автор, куратор, продажи;
- есть полноценный учебный сценарий: курс, модули, уроки, домашки, проверка, прогресс;
- есть понятная архитектура, чтобы команда могла развивать продукт без зависимости от ИИ.

## 2. Правило работы

Дальше идем строго фазами.

Правила:

- не распыляемся на новые большие блоки, пока не закрыт текущий;
- каждая фаза заканчивается критериями готовности;
- только после этого переходим к следующей;
- все крупные этапы фиксируются коммитом, пушем и записью в хронологию GitHub.

## 3. Что уже собрано

На текущий момент уже есть:

- `Next.js 15 + Prisma` foundation;
- Railway deployment;
- email/password auth;
- роли `ADMIN`, `AUTHOR`, `CURATOR`, `SALES_MANAGER`, `STUDENT`;
- admin/workspace shell;
- редактор курса с модульной структурой;
- block-based lesson editor;
- demo billing;
- homework foundation;
- student analytics foundation;
- invite-flow для команды;
- публичный лендинг, каталог, sign-in.

Это значит, что дальше мы не строим фундамент заново, а дожимаем продуктовые слои.

## 4. Главный порядок исполнения

### Фаза 1. Course Studio UX

Цель:
превратить редактор курса в реально удобный рабочий инструмент.

Что делаем:

- drag-and-drop уроков в левом дереве между модулями;
- drag-and-drop уроков внутри одного модуля;
- drag-and-drop модулей;
- быстрые действия прямо в дереве:
  - добавить модуль;
  - добавить урок;
  - переименовать;
  - удалить;
- cleanup editor:
  - меньше шумных подсказок;
  - меньше лишних карточек;
  - один понятный canvas по центру;
  - минимум secondary UI;
- cleanup video block:
  - один сценарий `вставить ссылку или загрузить файл`;
  - без перегруженных параллельных карточек.

Критерий готовности:

- структуру курса можно редактировать почти без прокрутки всей страницы;
- урок реально переносится мышкой между модулями;
- в дереве сразу видно актуальную структуру;
- экран не перегружен служебным текстом;
- методист может собирать курс без ощущения "формы из 10 блоков".

Статус:
`следующий активный этап`

---

### Фаза 2. Homework Review V2

Цель:
сделать домашки полноценным учебным workflow, а не просто формой сдачи.

Что делаем:

- очередь проверок с фильтрами:
  - новые;
  - на проверке;
  - на доработке;
  - принятые;
- история попыток одной домашки;
- нормальные комментарии куратора;
- явный статус у студента;
- возврат на доработку;
- автоматическое открытие следующего модуля после принятия;
- быстрый переход:
  - к студенту;
  - к уроку;
  - к курсу;
- улучшение student-side:
  - что именно сдавать;
  - что уже отправлено;
  - что вернул куратор.

Критерий готовности:

- куратор работает из одной очереди;
- студент понимает текущий статус без вопросов;
- домашка становится реальной частью учебного маршрута.

Статус:
`после Фазы 1`

---

### Фаза 3. Role-Specific Workspaces

Цель:
развести внутренний продукт по реальным ролям, а не только по permissions.

Что делаем:

- `AUTHOR workspace`:
  - только свои курсы;
  - быстрый вход в редактор;
  - минимум лишних admin-функций;
- `CURATOR workspace`:
  - домашки;
  - студенты;
  - прогресс;
  - проблемные точки;
- `SALES_MANAGER workspace`:
  - письма;
  - оплаты;
  - доступы;
  - лиды и ручные действия;
- role-specific dashboards вместо общего admin-overview.

Критерий готовности:

- каждая роль после логина видит "свой" продукт;
- не нужно ментально фильтровать чужие разделы.

Статус:
`после Фазы 2`

---

### Фаза 4. Student Analytics Dashboard

Цель:
сделать админке и кураторам понятную поведенческую аналитику студента.

Что делаем:

- карточка студента:
  - где остановился;
  - когда был последний вход;
  - какие уроки не досматривает;
  - какие домашки не сдает;
- аналитика по курсу:
  - точки отваливания;
  - паузы;
  - закрытия страницы;
  - недосмотры видео;
- триггеры риска:
  - не заходил N дней;
  - бросил урок на середине;
  - застрял перед домашкой;
- админский обзор по проблемным студентам.

Критерий готовности:

- по каждому студенту видно, где он потерялся;
- куратор и продажи видят, кому нужен возвратный контакт.

Статус:
`после Фазы 3`

---

### Фаза 5. Catalog / Checkout Polish

Цель:
довести продающий контур до ощущения готового продукта.

Что делаем:

- усиление карточек курсов;
- улучшение pricing presentation;
- cleaner checkout;
- больше proof и trust elements;
- более сильный mobile-pass;
- единый дизайн-язык между лендингом, каталогом и sign-in.

Критерий готовности:

- публичный слой можно показывать как продажный продукт, а не как демонстрацию.

Статус:
`после Фазы 4`

---

### Фаза 6. Реальные интеграции

Цель:
перевести критичные demo-контуры в production.

Что делаем:

- реальные платежи:
  - первый payment provider;
  - webhooks;
  - подтверждение оплаты;
- реальные письма:
  - `Resend`;
  - webhook statuses;
  - cron queue;
- video production layer:
  - provider lifecycle;
  - реальные статусы;
  - storage/processing pipeline.

Критерий готовности:

- продукт не только показывает сценарии, но и реально обслуживает продажи и коммуникацию.

Статус:
`после Фазы 5`

---

### Фаза 7. Files / Media / Submissions

Цель:
убрать зависимость от "ссылок вместо файлов" там, где это уже мешает работе.

Что делаем:

- файловое хранилище материалов уроков;
- хранение вложений домашек;
- более чистый сценарий для методистов и студентов;
- контроль типов и размеров файлов.

Критерий готовности:

- материалы и сдачи работают как часть платформы, а не как набор внешних ссылок.

Статус:
`после Фазы 6`

---

### Фаза 8. Final Design Pass

Цель:
свести весь продукт к одному сильному визуальному языку.

Что делаем:

- финальный pass по лендингу;
- финальный pass по catalog / checkout;
- финальный pass по course studio;
- финальный pass по workspaces;
- mobile consistency;
- пустые состояния, ошибки, onboarding states.

Критерий готовности:

- продукт выглядит как цельная платформа, а не как набор отдельно сделанных экранов.

Статус:
`после Фазы 7`

## 5. Что не делаем раньше времени

Пока не закрыта Фаза 1, не уходим глубоко в:

- новые маркетинговые блоки;
- дополнительную аналитику писем;
- сложные автоматизации;
- декоративные фичи;
- новые типы интеграций, если они не критичны для текущей фазы.

Смысл:
сейчас главный рычаг качества — это удобство сборки и сопровождения курсов.

## 7. Definition of Done для каждого пакета

Каждый пакет считается завершенным только если:

- сценарий работает в UI;
- `lint` проходит;
- `build` проходит;
- изменения запушены в GitHub;
- Railway обновлен, если фича влияет на живой стенд;
- в issue хронологии добавлен комментарий.

## 8. Формат следующих ходов

Каждый следующий этап ведем так:

1. Берем одну фазу.
2. Разбиваем на 1-3 исполнимых пакета.
3. Доводим пакет до рабочего состояния.
4. Фиксируем в GitHub и хронологии.
5. Только потом берем следующий пакет.
